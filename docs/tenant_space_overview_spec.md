# 企业空间总览页规格（V1）

## 1. 页面定位

- 页面名称：`空间总览`
- 建议路由：`/system/overview`
- 建议菜单位置：企业空间 `系统管理` 分组内，放在 `成员准入` 之前
- 可见角色：`owner`、`admin`
- 隐藏角色：普通企业成员、SaaS 管理员

### 1.1 页面目标

让企业管理员进入系统管理后，先一眼看到当前企业空间的：

- 人：现在有多少成员，最近有没有人在用
- 资产：数据项目、看板、术语、SQL 示例、自定义 Agent、嵌入应用是否已经沉淀
- 配置：空间是不是已经具备基本可用能力
- 待办：现在有哪些管理动作需要处理
- 动态：最近发生了什么关键变化

### 1.2 与“用量监控”的边界

`空间总览` 和 `用量监控` 必须明确分工：

- `空间总览`：看企业空间经营状态、配置完整度、管理待办
- `用量监控`：看请求、Token、任务、成功率、失败率、对账数据

`空间总览` 不展示以下指标，避免和 `用量监控` 冲突：

- 总 Token 消耗
- 请求数
- 成功数 / 失败数
- 成功率 / 失败率
- 任务数
- Token 分布 / Token 趋势


## 2. 信息架构

### 2.1 页面结构

页面建议分成 4 个层级：

1. 顶部头部区
2. 第一屏 KPI 总览区
3. 中部图表区
4. 底部待办与动态区

### 2.2 模块划分

#### A. 头部区

- 页面标题：`空间总览`
- 企业名：当前企业空间名称
- 时间范围切换：`近7天 / 近14天 / 近30天`
- 刷新按钮

说明：

- 时间范围只影响“成员活跃趋势”和“最近动态”
- 资产类统计不随时间范围变化

#### B. 第一屏 KPI 卡片区

建议 5 个卡片：

1. 成员总数
2. 近7天活跃成员数
3. 数据项目数
4. 已发布看板数
5. 待处理成员申请数

#### C. 中部图表区

建议 3 个核心模块：

1. 成员活跃趋势
2. 空间资产构成
3. 成员角色分布

#### D. 底部管理区

建议 2 个实用模块：

1. 待处理事项
2. 最近动态

## 4. 模块定义

## 4.1 KPI 卡片区

### 4.1.1 成员总数

- 说明：当前企业空间已加入且生效的成员总数
- 展示形式：数字卡片
- 点击动作：跳转到 `成员准入`

### 4.1.2 近7天活跃成员数

- 说明：近 7 天内至少登录过一次的企业成员数
- 展示形式：数字卡片 + 辅助文案
- 辅助文案：`较前7天` 的增减值可作为 V1.5，不强行首版上
- 点击动作：定位到 `成员活跃趋势`

### 4.1.3 数据项目数

- 说明：当前企业空间下的数据项目 / 数据源数量
- 展示形式：数字卡片
- 点击动作：跳转到 `项目管理`

### 4.1.4 已发布看板数

- 说明：当前企业空间内已创建并可访问的看板数量
- 展示形式：数字卡片
- 点击动作：跳转到 `数据看板`

### 4.1.5 待处理成员申请数

- 说明：等待企业管理员审核的加入企业申请数量
- 展示形式：数字卡片，数量大于 0 时高亮
- 点击动作：跳转到 `成员准入`


## 4.2 成员活跃趋势

### 4.2.1 模块目标

回答两个问题：

- 最近有没有人在持续使用这个企业空间
- 活跃是单点登录，还是多人持续使用

### 4.2.2 图表形式

- 主图：折线图或面积折线图
- 推荐使用现有 AntV `line`

### 4.2.3 系列定义

建议两条线：

- 活跃成员数
- 登录次数

如果首版想更克制，可以只保留一条 `活跃成员数`。

### 4.2.4 交互

- 切换 `近7天 / 近14天 / 近30天`
- Hover 展示当天活跃成员数和登录次数


## 4.3 空间资产构成

### 4.3.1 模块目标

回答：

- 这个企业空间只是“建了个壳”，还是已经有内容沉淀
- 内容主要沉淀在哪些模块

### 4.3.2 图表形式

- 推荐：横向条形图
- 推荐使用现有 AntV `bar`

理由：

- 类目名称长，条形图更稳定
- 用户更容易同时看“绝对数量”和“类型差异”
- 比饼图更适合这类后台配置型页面

### 4.3.3 资产类型

建议展示 6 类：

1. 数据项目
2. 已发布看板
3. 业务术语
4. SQL 示例
5. 自定义 Agent
6. 嵌入应用

### 4.3.4 点击动作

- 数据项目 -> `项目管理`
- 已发布看板 -> `数据看板`
- 业务术语 -> `术语配置`
- SQL 示例 -> `SQL 示例库`
- 自定义 Agent -> `自定义 Agent`
- 嵌入应用 -> `嵌入式管理`


## 4.4 成员角色分布

### 4.4.1 模块目标

回答：

- 当前空间的组织结构是否健康
- 是只有管理员，没有成员，还是已经形成协作

### 4.4.2 图表形式

- 推荐：环图
- 推荐使用现有 AntV `pie`

### 4.4.3 角色口径

只统计当前已加入企业的有效成员，不混入申请中人员。

角色分组建议为：

- 企业所有者（owner）
- 企业管理员（admin）
- 普通成员（member）

如想进一步简化，也可合并成：

- 管理员（owner + admin）
- 普通成员（member）

### 4.4.4 附加信息

环图旁边展示：

- 总成员数
- 管理员占比


## 4.5 待处理事项

### 4.5.1 模块目标

告诉管理员当前最需要动作的事情，避免进入系统管理后还要自己逐个菜单找。

### 4.5.2 展示形式

- 列表卡
- 每项包含：名称、数量/状态、跳转入口

### 4.5.3 首版建议项

#### 审批类

- 待处理加入企业申请

#### 配置缺口类

- 尚未配置数据项目
- 尚未沉淀业务术语
- 尚未沉淀 SQL 示例
- 尚未创建数据看板

#### 安全配置类

- 未开启强制 SSO（仅提醒，不作为错误）
- 会话超时时间未配置（仅提醒）

说明：

- “配置缺口” 数量通常展示为 `0/1` 状态项，而不是纯数值排行
- 待办列表要优先展示“真要处理”的动作，不要变成说明文案墙


## 4.6 最近动态

### 4.6.1 模块目标

回答：

- 最近谁做了什么
- 企业空间在往前推进，还是处于停滞状态

### 4.6.2 展示形式

- 时间线
- 最多展示最近 8~10 条

### 4.6.3 动态类型

建议只展示高价值动态，不直接把所有审计日志原样搬过来。

首版建议保留：

- 新成员加入企业
- 加入企业申请被批准 / 拒绝
- 新增数据项目
- 新增业务术语
- 新增 SQL 示例
- 新建自定义 Agent
- 新建嵌入应用
- 新建或发布看板
- 更新安全策略


## 5. 字段口径

## 5.1 顶部 KPI 字段

| 字段 | 含义 | 统计口径 | 时间范围 | 是否与用量监控冲突 |
| --- | --- | --- | --- | --- |
| member_total | 成员总数 | 当前企业空间内 `tenant_user.status=1` 且用户有效的成员数 | 实时 | 否 |
| active_members_7d | 近7天活跃成员数 | 近7天内有成功登录日志的去重成员数 | 近7天 | 否 |
| datasource_total | 数据项目数 | 当前企业空间下数据项目总数 | 实时 | 否 |
| dashboard_total | 已发布看板数 | 当前企业空间可访问的看板叶子节点数 | 实时 | 否 |
| pending_member_applications | 待处理成员申请数 | 当前企业空间待审核加入申请数 | 实时 | 否 |

## 5.2 图表字段

### 成员活跃趋势

| 字段 | 含义 | 统计口径 |
| --- | --- | --- |
| stat_date | 统计日期 | 按天 |
| active_member_count | 活跃成员数 | 当天有成功登录日志的去重成员数 |
| login_count | 登录次数 | 当天成功登录次数 |

### 空间资产构成

| 字段 | 含义 | 统计口径 |
| --- | --- | --- |
| asset_type | 资产类型 | datasource / dashboard / terminology / training / custom_agent / embedded |
| asset_label | 资产显示名 | 数据项目 / 已发布看板 / 业务术语 / SQL 示例 / 自定义 Agent / 嵌入应用 |
| asset_count | 资产数量 | 各模块当前企业空间总量 |

### 成员角色分布

| 字段 | 含义 | 统计口径 |
| --- | --- | --- |
| tenant_role | 企业角色 | owner / admin / member |
| role_label | 展示名 | 企业所有者 / 企业管理员 / 普通成员 |
| member_count | 数量 | 当前生效成员数 |

## 5.3 待处理事项字段

| 字段 | 含义 | 统计口径 |
| --- | --- | --- |
| pending_member_applications | 待处理加入申请 | `status=pending` 的 tenant application 数量 |
| empty_datasource | 未配置数据项目 | `datasource_total == 0` |
| empty_terminology | 未配置业务术语 | `terminology_total == 0` |
| empty_training | 未配置 SQL 示例 | `training_total == 0` |
| empty_dashboard | 未创建看板 | `dashboard_total == 0` |
| sso_not_enabled | 未开启强制 SSO | `sso_required == false` |
| session_timeout_missing | 未配置会话超时 | `session_timeout_minutes` 为空或 0 |

## 5.4 最近动态字段

| 字段 | 含义 |
| --- | --- |
| event_time | 事件时间 |
| event_type | 事件类型 |
| actor_name | 操作人 |
| target_name | 目标对象 |
| summary | 展示文案 |
| target_route | 点击跳转目标 |


## 6. 数据来源映射

## 6.1 现有接口可直接复用的数据

| 页面模块 | 可复用来源 | 备注 |
| --- | --- | --- |
| 待处理成员申请数 | `tenantApi.tenantApplications('pending')` | 已有 |
| 数据项目数 | `datasourceApi.list()` | 已有 |
| 术语数 | `professionalApi.getList(1, 1, params)` 的 `total_count` | 已有，但建议不要前端多次拼接口 |
| SQL 示例数 | `trainingApi.getList(1, 1, params)` 的 `total_count` | 已有，但建议聚合 |
| 自定义 Agent 数 | `promptApi.getList(1, 1, type, query)` 汇总 `total_count` | 已有，但要按类型聚合 |
| 嵌入应用数 | `embeddedApi.getList(1, 1, params)` 的 `total` | 已有 |
| 安全策略提醒 | `tenantApi.security()` | 已有 |

## 6.2 不建议前端硬拼的统计

以下数据虽然理论上能拆接口算出来，但不建议在前端自己拼：

- 成员总数
- 近7/14/30天活跃成员数
- 成员活跃趋势
- 成员角色分布
- 已发布看板数
- 最近动态

原因：

- 需要跨表聚合
- 需要统一 tenant 口径
- 需要减少页面首屏并发请求
- 需要让“总览”与“用量监控”保持口径一致

## 8. 首版落地建议

## 8.1 V1 必做

- 新增 `空间总览` 菜单
- 顶部 5 个 KPI
- 成员活跃趋势
- 空间资产构成
- 成员角色分布
- 待处理事项
- 最近动态
- 新增聚合接口 `GET /system/tenant/overview`

## 8.2 V1 可以先不做

- 环比、同比、趋势箭头
- 资产健康评分
- 看板点击热度排行
- 成员活跃排行榜
- 多企业切换视角


## 9. 前端实现建议

- 使用现有 AntV 图表封装，不新增新的图表技术栈
- 颜色风格保持与当前 `用量监控` 一致，但更偏“管理总览”而非“账单监控”
- 页面不做成营销式大卡片堆叠，保持后台管理面的安静、清晰、可扫描
- 待办项和最近动态必须可点击跳转，不要只做成静态展示


## 10. 结论

这页的正确定位不是“另一个监控页”，而是：

`企业空间总览 = 人 + 资产 + 配置 + 待办 + 动态`

而：

`用量监控 = 请求 + Token + 任务 + 对账`

只要按这个边界落地，两页就不会打架，而且用户会非常容易理解。
